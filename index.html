Locomotive - Technical Deep Dive

Architecture

Locomotive deploys as a sidecar container in an existing Railway project. It leverages the internal Railway GraphQL API to subscribe to deployment log streams. The user configures the application using a Railway template, and environment variables. Locomotive is pointed at an external ingest URL, where log lines are sent using a specified format and processed by the consumer.

Key Challenges

  • Handling deployments updating and restarting gracefully without dropping log lines.
  • Reconstructing Railway's fragmented log lines from the GraphQL API into full log lines useable by external consumers.
  • Packaging unstructured text logs into objects useable by other platforms.

Tech Stack Details

Written in Go. Heavy use of Goroutines & channels for handling multiple internal tasks at once, including incoming log lines and forwarding logs to external platforms. Locomotive is packaged as a Docker container and deployed using a Railway template. GraphQL is used to subscribe to incoming logs, events, and other important details about monitored deployments.

Achievements

Received multiple sponsorships from Railway to expand the feature set and improve reliability. Shipped well over 1m+ logs with 100% uptime. Deployed in 30+ user applications.

Scavenger - Technical Deep Dive

Architecture

Scavenger is a Dockerized self-hostable platform that allows developers to quickly create web scraping pipelines. Users create workflows by describing in english what content they want, and then defining a structured schema for the data to be packaged in. Each web scraping job uses an LLM to analyze the page content & extract the requested information. Each unit of data is streamed over websocket, where consuming applications can connect and retrieve data in real time. Built for BeaverHacks in 24 hours.

The technical implementation involved a monolithic Golang application that used dependency injection to spin up workflow containers on the seleted cloud platform (or locally via Docker + self-orchestration). Each workflow container is written in a multithreaded Python application using Crawl4AI to scrape and package the page content. For each workflow, a new private websocket endpoint (only accessible via API key) is spun up that consumers connect to. Workflows are scheduled using user-specified cron strings to run at intervals.

Key Challenges

  • Integrating the GCP cloud compute API & getting containers to reliably spin up and pass health checks.
  • Implementing concurrency reliably in Python workflows.
  • Implementing reliable Websocket connections & statistics tracking for workflows

Tech Stack Details

Written in Go. Used Gorilla websockets for websocket connections, and listened concurrently for both workflow & consumer connections. Implemented a websocket reverse proxy between the workflow & the user for streaming. Used Docker for deployment, and MongoDB for workflow & document storage. Google Cloud Run API for GCP integration, and Docker CLI package for local orchestration. Used Pulumi for user-facing IaC to simplify deployment on GCP.

Achievements

Built in under 24 hours for BeaverHacks with a high level of polish. Collaborated with a team of 3 other developers & wrote high-quality and maintainable code that allowed easy expansions after the event.

FleetOS - Technical Deep Dive

Architecture

FleetOS is an open-source telmatics device used for fleet tracking. It integrates with a backend ingest platform over MQTT. The device uses a GNSS chip, a cellular modem, and an STM32 to track and transmit fleet locational data. The backend uses Golang and EMQX for ingress, and stores data in a TimescaleDB database for chronological data. The dashboard allows for fleet managers to view locations, introduce geofences, and manage devices remotely. An authentication strategy using pre-baked certificates is used to ensure no spoofing or interception occurs between the device and the server.

Key Challenges

  • Integrating the GNSS chip. It requires filtering, impedance matching, and knowledge of various communication protocols and antenna considerations.
  • Managing possible frequency collisions between GNSS and cellular.
  • Keeping devices secure & designing an authentication strategy.

Tech Stack Details

Device uses a simple STM32 F-series MCU, a Quectel NEO-M9N GNSS chip, and a Quectel BC65PB-04-STD cellular modem. Backend is written in Go, and MQTT is handled via an EMQX broker.

Achievements

Learned about managing frequency & RF conflicts, implementing complex RF circuitry, and security in embedded devices. Worked with and led a team of 4 others to plan and execute project requirements.

OSUSat - Technical Deep Dive

Architecture

OSUSat is a student CubeSat project I started and lead at Oregon State. Right now it's a prototype climate science satellite with the goal of launching it into low Earth orbit in the coming yeras. It's a modular system, allowing subsystems, boards, and firmware components to be swapped out quickly. It's based on four subsystems: the EPS (power distribution board), OBC (main system control board), payload (scientific instrument), and comms (downlink). Each of these subsystems requires its own complex PCB and firmware project.

Key Challenges

  • Breaking down, designing, and implementing a complex system within a short timespan.
  • Leading a team of 5-6 engineers to design a cohesive system in an organized fashion.
  • Creating robust but simple standards and philosophies for design & implementation.
  • Implementing flight-ready electronics in a student environment.

Tech Stack Details

Each board (bar comms for the prototype) relies on an STM32 microcontroller. Two instances of the STM32H7 family and one instance of the STM32L4 family.

The EPS contains complex analog circuitry for hardware current monitoring & load shedding, as well as a complex C firmware project to manage power policy & system state. Individually controlled power rails are toggled via a load switch, both by monitoring circuitry and firmware.

The OBC controls broad mission logic by performing health checks, monitoring subsystems, and using our custom packet standard & codec to request and command other subsystems. This subsystem also manages all coordination with the ground, meaning it supports manual maintenance/probing, telemetry packaging, and more. It also uses QSPI and PSRAM to buffer data from other subsystems, and can move the system into "safe mode" in the case of an emergency.

The payload board is the science instrument on the satellite, and its goal is to take multi-spectral images, and run processing algorithms on them to monitor climate health over time. It uses algorithms like NDVI, NDCI, and more to monitor vegetation health and oceanic algae blooms, which change over time with the climate. It's based on an STM32H7, and is responsible for collecting information, buffering, compressing, and coordinating with the OBC to transport it back to the ground.

The comms system is the simplest board (for the prototype), just boasting an RFM69HCW LoRa transmitter. It exposes an SPI interface for the OBC to control for downlink purposes.

Our general design philosophy is centered around safety and modularity. From the structural and hardware side to the firmware side, the entire system is easily able to swap in and out. Using a modular clamping desing, boards are easily replaced simply by unbolting and bolting them in. The firmware uses build-time dependency injection to conditionally link components based on system configuration and testing purposes. The firmware is also designed to be testable using the HITL philosophy, where some or all of the hardware components are simulated to test the firmware independently.

I did an in-depth writeup on the firmware architecture that goes into far greater detail about the reasoning and structure that can be found here. I would highly encourage you to read it if you're interested in the nitty-gritty.

Achievements

Collaborated with a team to design and implement a complex system in a short time frame, while still maintaining high levels of quality, documentation, and robustness. Learned about how to design space hardware, and implemented industry-standard practices for safety in many areas of the system.

Switchyard - Technical Deep Dive

Architecture

Switchyard is a feature-complete, pluggable toolkit for infrastructure management on Railway. It's deployed simply into an existing Railway project via a Railway template, and exposes multiple services that enable production-grade scaling, testing, and observability in Railway apps, without code changes. Switchyard was the 2025 Railway Hackathon 1st place winner!

Switchyard has autoscaling, job scheduling, user-defined workers, feature flags, and incident reporting. All of this is managed through a simple web dashboard. Read about the features in more detail in the whitepaper.

Key Challenges

  • Making scaling decisions based on a wide history of input data for multiple deployments.
  • Interacting with the Railway GraphQL API in many contexts.
  • Managing configuration across deployments.

Tech Stack Details

Switchyard is implemented in five different microservices (all implemented in Go) with various supporting services. The first service is the autoscaling service, which uses a sliding window algorithm to continuously look at deployment metrics to make decisions. Its algorithm is robust, and can detect spiking, emergency scenarios, general sustained load, and low use periods to make horizontal scaling decisions automatically.

The next microservice is the job scheduling service, which uses RabbitMQ, Redis, and custom job workers to offload work from the main system. It allows users to write their own custom job handlers, and Switchyard provides documentation on how to adhere with its deduplication and reliability measures. Users deploy their workers as normal Railway services, and configure them in the dashboard to handle tasks. When the main system sends a request to offload some work, the scheduler looks at the request and queues a job receipt for the handler to work on. In combination with the autoscaling algorithm, it allows the worker pool to autoscale with demand.

The feature flags microservice allows applications to conditionally expose certain features on their platform to users. The dashboard allows administrators to create feature flags like "show X page to internal testers, but show Y page to normal users." As applications hit these code paths where decisions are made, they simply psas the user context to Switchyard, which automatically determines which path to take.

The incident reporting service monitors both deployment status, as well as uses an internal locomotive deployment to monitor for errors. Users subscribe to interested status changes (e.g. HEALTHY -> CRASHED), which automatically are streamed to the dashboard, as well as forwarded to incident channels like Slack or Discord. Additionally, Locomotive is bundled with Switchyard to allow users to monitor and ingest logs.

The last microservice is the configurator, which allows internal applications to configure and redeploy Switchyard services at runtime to meet system needs. It also allows applications to request changes to environment variables for user apps on the fly.

Switchyard services communicate over Railway's internal networking, which keeps everything airgapped from external services. The only exposed service is the dashboard, which is normally only exposed for short periods using a TCP proxy.
